Fine-Tuning Large Language Models: A Practical Guide

How to adapt a 7B or 8B parameter model to your domain using LoRA and Unsloth — without needing a data centre

Alex Rivera Nov 15, 2024 12 min read
  • Machine Learning
  • LLM
  • Python
  • Deep Learning
  • NLP

Large language models have transformed what software can do. But out-of-the-box, a general-purpose model often falls short when you need consistent tone, domain-specific vocabulary, or structured output formats. Fine-tuning bridges that gap — and in 2024 it has never been more accessible.

In this article I'll walk through a complete fine-tuning pipeline using Unsloth, Hugging Face TRL, and the Alpaca dataset. By the end you'll have a LoRA-adapted Llama 3 8B model running on a single consumer GPU.

Why Fine-Tune at All?

Prompt engineering solves a lot of problems cheaply. Before committing to a fine-tuning run, ask yourself whether a well-crafted system prompt or few-shot examples already get you where you need to go. Fine-tuning earns its cost when:

  • You need the model to reliably output a very specific format (JSON, XML, custom schema).
  • Your domain has terminology the base model consistently misuses or hallucinates.
  • You are running thousands of inferences daily and want to reduce prompt length (and therefore cost).
  • You need to embed proprietary knowledge that cannot be sent to a third-party API.

Understanding LoRA

Full fine-tuning updates every parameter in the model — all 8 billion of them for Llama 3 8B. At fp16 that is 16 GB just to store the weights, and you need roughly 3× that for gradients and optimiser state. Most practitioners do not have that hardware.

Low-Rank Adaptation (LoRA), introduced by Hu et al. in 2021, sidesteps this by freezing the original weights and injecting small trainable matrices into each attention layer. If the original weight matrix is W ∈ ℝ^(d×k), LoRA adds ΔW = BA where B ∈ ℝ^(d×r) and A ∈ ℝ^(r×k) with rank r ≪ min(d, k). Typically r=8 or r=16.

"The key insight is that the weight updates during fine-tuning have a low intrinsic rank, even though the weight matrices themselves are full rank."

— Hu et al., LoRA: Low-Rank Adaptation of Large Language Models (2021)

QLoRA extends this by loading the base model in 4-bit NormalFloat (NF4) quantisation, cutting VRAM requirements by a further 75%. This is what makes fine-tuning a 7B or 8B model feasible on a single 12 GB consumer GPU.

Setting Up the Environment

First, install the required packages. Unsloth handles the Triton/CUDA kernels that make LoRA training 2× faster than vanilla PEFT:

pip install "unsloth[colab-new] @ git+https://github.com/unslothai/unsloth.git"
pip install --no-deps trl peft accelerate bitsandbytes

Verify that your GPU is detected and that bfloat16 is supported:

import torch
print(torch.cuda.get_device_name(0))
print("bfloat16 supported:", torch.cuda.is_bf16_supported())

Loading the Base Model

Unsloth provides patched versions of popular architectures. Loading a 4-bit model takes roughly 10 seconds on a modern GPU: 

from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="unsloth/llama-3-8b-bnb-4bit",
    max_seq_length=2048,
    dtype=None,          # auto-detect
    load_in_4bit=True,
)

The dtype=None argument lets Unsloth choose between float16 and bfloat16 based on GPU capability. Ampere and later (RTX 30xx, A100) support bfloat16, which is more numerically stable for training.

Attaching LoRA Adapters

The get_peft_model call freezes the base weights and injects trainable adapter matrices. The target modules listed below are the attention and feed-forward projections — the layers that matter most for task adaptation: 

model = FastLanguageModel.get_peft_model(
    model,
    r=16,
    target_modules=[
        "q_proj", "k_proj", "v_proj", "o_proj",
        "gate_proj", "up_proj", "down_proj",
    ],
    lora_alpha=32,
    lora_dropout=0.0,
    bias="none",
    use_gradient_checkpointing="unsloth",
)

Setting lora_dropout=0.0 is recommended by the Unsloth authors — for most fine-tuning workloads dropout in LoRA layers hurts convergence more than it helps generalisation. Gradient checkpointing with the Unsloth backend reduces VRAM by ~30% at the cost of a small recomputation overhead.

Preparing the Dataset

We'll use the cleaned Alpaca dataset from yahma/alpaca-cleaned. The dataset contains around 52,000 instruction-following pairs in the schema {instruction, input, output}. We convert each row to a single formatted string: 

ALPACA_PROMPT = """Below is an instruction that describes a task.
Write a response that appropriately completes the request.

### Instruction:
{instruction}

### Input:
{input}

### Response:
{output}"""

def format_alpaca(examples):
    texts = []
    for inst, inp, out in zip(
        examples["instruction"],
        examples["input"],
        examples["output"]
    ):
        texts.append(ALPACA_PROMPT.format(
            instruction=inst, input=inp, output=out
        ) + tokenizer.eos_token)
    return {"text": texts}

from datasets import load_dataset
dataset = load_dataset("yahma/alpaca-cleaned", split="train")
dataset = dataset.map(format_alpaca, batched=True)

Training

SFTTrainer from TRL handles the supervised fine-tuning loop. Key hyperparameters:

Parameter Value Rationale
learning_rate 2e-4 Standard starting point for LoRA on instruction data
lr_scheduler_type cosine Smooth decay avoids loss spikes at end of training
optim adamw_8bit 8-bit Adam cuts optimiser state memory by 75%
per_device_train_batch_size 2 Safe default for 12 GB VRAM with seq_len=2048
gradient_accumulation_steps 4 Effective batch size of 8 without extra VRAM

Saving and Exporting

After training, save the LoRA adapters. These are small — typically 50–200 MB depending on rank — and can be shared or version-controlled independently of the base model: 

model.save_pretrained("./my-llama3-adapter")
tokenizer.save_pretrained("./my-llama3-adapter")

To merge the adapters into the base weights and export as a standalone 16-bit model:

model.save_pretrained_merged(
    "./my-llama3-merged",
    tokenizer,
    save_method="merged_16bit"
)

For local inference with llama.cpp or Ollama, export to GGUF:

model.save_pretrained_gguf(
    "./my-llama3-gguf",
    tokenizer,
    quantization_method="q4_k_m"
)

Results and Next Steps

After 60 steps (about 4 minutes on an RTX 3090) the training loss drops from ~2.1 to ~1.4 and the model responds noticeably more consistently to instruction-style prompts. A full epoch of Alpaca typically brings loss to ~0.9.

From here, the natural next steps are:

  1. Swap the dataset for your own domain-specific corpus.
  2. Experiment with rank (r=32 or r=64) for tasks requiring more capacity.
  3. Evaluate with a held-out test set using MT-Bench or domain-specific metrics.
  4. Serve the merged model with vLLM or Ollama behind a simple API.

Fine-tuning has moved from research infrastructure to a weekend project. The combination of 4-bit quantisation, LoRA, and optimised training kernels means that meaningful model adaptation is now within reach for any practitioner with a single modern GPU.
